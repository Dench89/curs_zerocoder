# KODA.md

## Назначение файла

Этот файл содержит инструкционный контекст для будущей работы AI-помощника с репозиторием `curs_zerocoder`. Его следует использовать как краткую карту проекта: что здесь находится, как запускать код, какие соглашения уже видны из структуры и документации.

## Обзор проекта

`curs_zerocoder` — учебный Python-проект для изучения базовых инструментов разработки: Python, виртуальных окружений, Git, терминальных команд и работы с AI-помощником Koda CLI.

Текущее состояние проекта минимальное:

- основной исполняемый файл: `main.py`;
- директория для будущего исходного кода: `src/`;
- директория для будущих тестов: `tests/`;
- директория с учебными справочными материалами: `docs/`;
- файл зависимостей: `requirements.txt`;
- README с базовыми Git-командами и правилом организации окружения.

Проект пока не содержит полноценной архитектуры приложения. В `main.py` находится простой пример запуска Python-кода: вывод строки `Hello world`.

## Ключевые файлы и директории

### `main.py`

Точка входа текущего учебного проекта. Сейчас содержит минимальный пример:

```python
print("Hello world")
```

При развитии проекта этот файл можно использовать как простой запускной сценарий либо заменить на импорт логики из `src/`.

### `README.md`

Краткое описание учебного проекта. Зафиксировано основное правило организации Python-проектов:

> Один проект = одна папка = одно виртуальное окружение (`.venv`).

Также указаны базовые команды Git: `git status`, `git add`, `git commit`, `git push`, `git log`.

### `requirements.txt`

Файл сейчас содержит не список зависимостей, а учебное пояснение о назначении `requirements.txt`.

Важно: перед использованием команды `pip install -r requirements.txt` файл нужно привести к стандартному формату, где каждая строка содержит пакет, например:

```text
requests==2.32.3
pytest==8.3.0
```

Пока реальных внешних зависимостей в проекте не обнаружено.

### `src/`

Папка зарезервирована для основного исходного кода. На момент анализа пустая.

Рекомендуемое использование при дальнейшем развитии:

- размещать бизнес-логику и модули приложения в `src/`;
- не раздувать `main.py`, а использовать его как тонкую точку входа;
- группировать код по назначению: модели, сервисы, работа с файлами, CLI и т.п.

### `tests/`

Папка зарезервирована для тестов. На момент анализа пустая.

Если будут добавлены тесты, предпочтительно хранить их здесь и называть файлы по распространённому шаблону:

```text
test_*.py
```

### `docs/`

Директория с учебными справочниками.

- `docs/PYTHON.md` — команды Python, `venv`, `pip`, проверка окружения и работа с пакетами.
- `docs/GIT.md` — команды Git: статус, коммиты, история, ветки, восстановление, сравнение, настройки.
- `docs/LINUX.md` — базовые команды Linux: навигация, файлы, процессы, сеть, службы, архивы.
- `docs/HELP/gitignore_help` — заметка о необходимости `.gitignore` и исключении виртуальных окружений, кэшей и локальных данных.
- `docs/HELP/KODA CLI` — справочник по установке и запуску Koda CLI.
- `docs/HELP/Итеративный подход` — описание итеративного подхода к разработке: ТЗ, декомпозиция, генерация, проверка, улучшение.

## Сборка, запуск и тестирование

### Создание виртуального окружения

Для Windows PowerShell:

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
```

Альтернативно через Python Launcher:

```powershell
py -3 -m venv .venv
.venv\Scripts\Activate.ps1
```

### Запуск проекта

```powershell
python main.py
```

Ожидаемый текущий вывод:

```text
Hello world
```

### Установка зависимостей

В текущем состоянии `requirements.txt` не является корректным списком пакетов, поэтому команду ниже следует использовать только после приведения файла к стандартному формату:

```powershell
pip install -r requirements.txt
```

Если внешних зависимостей нет, файл можно оставить пустым или обновить после установки нужных пакетов:

```powershell
pip freeze > requirements.txt
```

## Правила разработки

### Общий подход

Проект учебный, поэтому предпочтителен итеративный стиль разработки:

1. Сформулировать небольшую задачу или техническое задание.
2. Разбить задачу на маленькие шаги.
3. Реализовать один шаг.
4. Запустить код и проверить результат.
5. Улучшить решение и перейти к следующей итерации.

### Организация Python-кода

Рекомендуемые правила для дальнейшего развития:

- держать один проект в одной папке;
- использовать отдельное виртуальное окружение `.venv` для проекта;
- основную логику размещать в `src/`;
- `main.py` использовать как точку входа;
- тесты хранить в `tests/`;
- не добавлять в репозиторий виртуальное окружение, кэши и локальные данные.

### Git-практики

Перед коммитом рекомендуется выполнять:

```powershell
git status
git diff
```

Обычный цикл работы:

```powershell
git status
git add .
git commit -m "Описание изменений"
git push
```

Для просмотра истории:

```powershell
git log --oneline --graph --decorate --all
```

### `.gitignore`

В репозитории есть справка `docs/HELP/gitignore_help`, но файл `.gitignore` в корне при анализе не обнаружен.

Рекомендуется добавить `.gitignore` и исключить минимум:

```gitignore
.venv/
venv/
__pycache__/
.pytest_cache/
*.pyc
```

Также следует исключать локальные JSON-файлы с реальными или персональными данными, если такие появятся.

## Рекомендации для будущих AI-взаимодействий

При работе с этим проектом следует:

- отвечать и документировать изменения на русском языке, если пользователь не попросил иначе;
- учитывать учебный характер проекта и объяснять решения простыми формулировками;
- не предполагать наличие зависимостей, пока они не зафиксированы в корректном `requirements.txt`;
- перед запуском команд установки учитывать, что текущий `requirements.txt` требует исправления;
- развивать проект маленькими итерациями;
- при добавлении Python-кода сначала определить, должен ли он находиться в `main.py` или в `src/`;
- при добавлении тестов использовать `tests/`;
- сохранять простую структуру и не усложнять архитектуру без необходимости.
